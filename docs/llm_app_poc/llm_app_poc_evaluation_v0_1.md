# LLMアプリ開発PoC 評価設計 v0.1

## 1. 評価の目的

このPoCでは、LLMアプリが単に文章を生成するだけでなく、実務で使える開発支援アウトプットを安定して生成できるかを評価する。

## 2. 評価対象

評価対象は以下。

- 要件要約
- 仕様整理
- 受け入れ条件
- Jiraチケット分解
- 実装方針
- レビュー観点
- リスク・確認事項

## 3. サンプル入力 1: プロフィール画像変更

```text
ユーザーがプロフィール画像を変更できるようにしたい。
画像は5MBまで、jpg/png対応。
変更後は即時反映したい。
失敗時にはエラーメッセージを表示したい。
```

期待する観点:

- ファイルサイズ制限が仕様に入っている
- jpg/png制限が仕様に入っている
- 成功時の即時反映が受け入れ条件に入っている
- 失敗時のエラー表示が受け入れ条件に入っている
- フロントエンド、バックエンド、テストのチケットに分解されている

## 4. サンプル入力 2: 検索フィルタ追加

```text
商品一覧画面に価格帯フィルタを追加したい。
最低価格と最高価格を入力できる。
検索条件はURLに反映して、ページを再読み込みしても維持したい。
不正な価格範囲の場合は検索しない。
```

期待する観点:

- 最低価格・最高価格の入力仕様が含まれる
- URLクエリへの反映が含まれる
- 再読み込み後の状態復元が含まれる
- 不正入力時のバリデーションが含まれる
- UI、検索ロジック、テストのタスクに分解されている

## 5. サンプル入力 3: 通知設定

```text
ユーザーがメール通知のオンオフを設定できるようにしたい。
通知種別はキャンペーン、セキュリティ、メンテナンスの3種類。
セキュリティ通知はオフにできない。
設定変更後は保存完了メッセージを表示したい。
```

期待する観点:

- 3種類の通知種別が仕様に入っている
- セキュリティ通知をオフにできない制約が含まれる
- 保存完了メッセージが受け入れ条件に含まれる
- バリデーション観点がレビュー観点に含まれる

## 6. 採点基準

各サンプル入力に対して、以下の5段階で評価する。

| 点数 | 基準 |
|---|---|
| 5 | 実務でほぼそのまま使える。重要な要件が網羅され、チケット粒度も適切 |
| 4 | 概ね使える。一部補足すれば実務投入可能 |
| 3 | 大枠は合っているが、抜け漏れや粒度の粗さがある |
| 2 | 一部使えるが、重要要件の抜け漏れが目立つ |
| 1 | 実務利用が難しい |

## 7. 重点評価項目

- 要件網羅性
- 具体性
- チケット分解の妥当性
- 受け入れ条件の検証可能性
- レビュー観点の実務性
- リスク抽出の妥当性
- JSON構造の安定性
